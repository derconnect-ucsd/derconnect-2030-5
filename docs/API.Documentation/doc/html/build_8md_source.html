1 Build System
2 ============
3 
4 `build.sh` is a Bash based build system/script, the purpose of the script is to
5 simplify C based cross platfom developement using gcc. While the script shares
6 features of a build system it is not intended as a replacement for existing
7 build systems such as `make` and the 'auto'-tools (`automake`/`autoconf`) whose
8 scope and application are much broader. The scope here is limited to simply
9 compiling applications and libraries that are part this framework using gcc as
10 the compiler and Bash as the shell/scripting environment.
11 
12 Bash was chosen as the language because of it's widespread use as a scripting
13 language and shell environment and it's use in configuring software (including
14 its use in build systems). The features of the Bash language (arrays,
15 introspection of variables) make it well suited to describe build targets, and
16 the flexibility of using a scripting language makes it possible to extend the
17 build system, for example defining custom build commands. These features are
18 more than adequate in addressing the current needs of the framework and future
19 needs as the framework is ported to other systems.
20 
21 Targets
22 -------
23 
24 Running the build script with no arguments will attempt to build the targets
25 specified in `targets.sh`. Within `targets.sh` the variable `targets` should
26 be defined, this variable is an array of targets to be built:
27 
28  targets=( ... )
29 
30 Each target can be either:
31 
32 - A subdirectory with its own `targets.sh`
33 - A target with it's own build command
34 - A C based target (executable, object, or shared library)
35 - A sublist of targets that contains any of these possibilities
36 
37 Targets can also be specified on the command line, in this case the file
38 `targets.sh` is still read and processed but the command line arguments become
39 the list of targets.
40 
41 Subdirectory Target
42 -------------------
43 
44 The subdirectory target is a subdirectory off of the main project directory
45 with its own `targets.sh`. Building a subdirectory target is equivalant to
46 changing to that directory and executing the build script. This feature is
47 used to define subprojects, these can be worked on independently of the main
48 project.
49 
50 Build Command Target
51 --------------------
52 
53 The build command target is a target that is "built" by executing a Bash
54 function. The name of this function is the target name with name `_build`
55 appended to it. For example, if a function is defined within `targets.sh` as:
56 
57  hello_build () { echo "Hello World"; }
58 
59 Then this target can be built by executing the command:
60 
61  ./build.sh hello
62 
63 C Based Targets
64 ---------------
65 
66 C based targets can be either an executable, object, or shared library. For a
67 given target name `target`, the following variables can be defined:
68 
69 - `target_flags` - compiler flags used to compile the target
70 - `target_libs` - libraries to be linked with the target
71 - `target_objects` - objects to be linked with the target
72 
73 The type of the target is determined by its name, targets ending in `.o` are
74 compiled objects, targets of the form `lib{target}.so` are shared library
75 targets, targets without either extensions are executable targets. In all cases,
76 the `_flags`, `_libs`, and `_objects` variables will have the target name
77 without the extensions, so that in compiling `libtest.so`, the variables
78 `test_flags`, `test_libs` and `test_objects` will be used.
79 
80 Only a single source file is used as the input to compile a target, its name is
81 the same as the target name with the ".c" extension used instead of the target
82 extensions. So, to produce `target`, `target.o`, or `libtarget.so` (an
83 executable, object, or shared library) the file `target.c` is used as the
84 input file.
85 
86 Sublist Targets
87 ---------------
88 
89 The final type of targets are sublist targets, these are Bash arrays that can
90 contain any of the prior types of targets. To create a sublist target, simply
91 create a Bash array and include it in the list of targets:
92 
93  sublist=( ... )
94  targets=( ... sublist ... )
95 
96 The `_flags, `_libs`, and `_objects` variables can be defined for a sublist
97 target:
98 
99  sublist_flags=( ... )
100  sublist_libs=( ... )
101  sublist_objects=( ... )
102 
103 When this occurs the targets included in the sublist will inherit these
104 attributes. This provides a useful way to group together executables or other
105 targets that share common attributes such as the compiler flags used and
106 libraries linked.
107 
108 Build Types
109 -----------
110 
111 The following build types can be specified on the command line:
112 
113 - `debug` - Compile targets with debugging symbols included and no
114  optimization.
115 - `static` - Use static linking.
116 - `cross` - Use the cross compiler instead of the default compiler.
117 
118 With no arguments specified, the default build type is optimized code
119 generation. Any of these build types can be combined, for example running the
120 command:
121 
122  ./build.sh debug cross
123 
124 Will use the cross compiler and compile the targets for debugging.

126 Dependencies and Rebuilding
127 ---------------------------
128 
129 Changes to the source files are frequently made in the course of development,
130 and as it so happens, the compiled targets (executables, objects, and shared
131 libraries) become out of date or unsyncronized with the source files. In such
132 cases a rebuild is required and it is desirable to rebuild only those files
133 that require it. In order to do that, we need to know what files a target
134 depends on. If any of these files are newer than the target then a rebuild is
135 required.
136 
137 In compiling a C source file, there are two kinds of dependencies, source file
138 dependencies created by including other source files (`#include` directives),
139 and binary dependencies that are implied through linking. The latter
140 dependencies are explicit within the build system, specified by listing the
141 linked objects required to build a target, the former can be discovered by the
142 gcc compiler itself using the `-MM` option. Before compiling a source file, the
143 build system checks a dependency file, one generated by running gcc with the
144 `-MM` option. From that listing of files, the build system can decide whether to
145 rebuild a target or not.
146 
147 The source dependency listings are only created the first time a target is
148 compiled. Because source files can sometimes change their source dependencies
149 (by adding or removing `#include` directives), the dependency files themselves
150 sometimes need to regenerated, this can be done with the "dep" command:
151 
152  ./build.sh dep
153 
154 Finally, a change in build type will require the targets to be rebuilt, for
155 this purpose the build system stores the set of build arguments and compares
156 this list with the build arguments specified on the command line. A change in
157 arguments will force a rebuild for all the targets.
